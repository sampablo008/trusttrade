# Supabase Setup

Step-by-step. No guessing.

## What exists in repo now

These SQL files are ready:

1. `supabase/migrations/0001_init.sql`
2. `supabase/migrations/0004_rls.sql`
3. `supabase/sql/110_promote_admin.sql`
4. `supabase/sql/120_verify_setup.sql`

These are the SQL files to run on Supabase for the current repo state.

## Step 1: Create projects

Create two Supabase projects:

1. `trustpro-dev`
2. `trustpro-prod`

Save these values for each project:

1. Project URL
2. Project ref
3. Anon key
4. Service role key
5. Database password

## Step 2: Add env vars

Create local env files with these keys:

```bash
NEXT_PUBLIC_SUPABASE_URL=
SUPABASE_ANON_KEY=
SUPABASE_SERVICE_ROLE_KEY=
```

## Step 3: Fix local CLI first

Current repo note: the `supabase` npm package was installed, but pnpm skipped build scripts, so the CLI binary is not usable yet.

Fix one of these ways:

1. Approve pnpm builds, then reinstall the package.
2. Install the official standalone Supabase CLI on your machine.

After that, verify:

```bash
supabase --version
```

## Step 4: Link local repo to project

Run:

```bash
supabase login
supabase link --project-ref <YOUR_DEV_PROJECT_REF>
```

## Step 5: Start local Supabase

Run:

```bash
supabase start
```

If you want local DB reset from migrations later:

```bash
supabase db reset
```

## Step 6: Run schema SQL

Run these SQL files in this exact order.

### Option A: Supabase SQL Editor

Open the dev project SQL Editor and run:

1. Contents of `supabase/migrations/0001_init.sql`
2. Contents of `supabase/migrations/0004_rls.sql`

### Option B: Supabase CLI

After the CLI works:

```bash
supabase db push
```

If you want a clean rebuild:

```bash
supabase db reset
```

## Step 7: Create first admin user

Do not insert into `auth.users` manually.

Create the first user from one of these:

1. Supabase Dashboard → Authentication → Users → Create user
2. A temporary login/signup flow
3. Supabase Admin API

After the auth user exists, run:

1. `supabase/sql/110_promote_admin.sql`

Replace:

```sql
YOUR_ADMIN_EMAIL_HERE
```

with the real email before running it.

## Step 8: Verify setup

Run:

1. `supabase/sql/120_verify_setup.sql`

This checks:

1. `app_config` seed row exists
2. `handle_new_user` trigger exists
3. RLS is enabled on the current core tables
4. policies were created

## Step 9: Expected current outcome

After running the current SQL:

1. Core tables from Sprint 0 exist
2. `handle_new_user` auto-creates `profiles` and `user_balances`
3. RLS is active on the current public tables
4. `is_admin()` exists
5. one seeded `app_config` row exists

## Not included yet

Not ready yet in repo:

1. invitation tables and functions
2. wallet/deposit/withdraw tables
3. referral tables
4. bonus ticket tables
5. higher timeframe candle tables
6. local Supabase config generated by a working CLI

Those will land in later phases and should not be expected from the current SQL.
